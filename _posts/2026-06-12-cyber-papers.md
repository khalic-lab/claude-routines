- **`vm2` Node.js sandbox — a cluster of CVSS 10.0 escape bugs, fix in 3.11.4.** Five separate critical CVEs land against `vm2`, a widely embedded library meant to run untrusted JavaScript safely inside Node.js (a "sandbox" — a walled-off interpreter). Each is a *sandbox escape*: attacker-controlled code inside the box reaches the host and runs arbitrary commands. The bypasses are textbook JS-engine prototype gymnastics — e.g. **CVE-2026-47131** (CVSS 10.0) chains `Buffer.call.call` with a `__lookupGetter__`/`__lookupSetter__` trick and Node's `ERR_INVALID_ARG_TYPE` error to recover the host's `TypeError` constructor and climb out; **CVE-2026-47137** (10.0) exploits a strict-equality (`=== false`) check that the fix for an *older* CVE introduced, so the patch itself reopened the hole; **CVE-2026-47140** (10.0) is a denylist that forgot `process` and `inspector/promises`; **CVE-2026-47210** (9.8) escapes via the WebAssembly JSPI promise-suspension API. Why it matters: `vm2` sits under many "run this snippet" features (online IDEs, low-code tools, serverless eval), so a 10.0 escape there is a full-host RCE (remote code execution — run any command) wherever it's deployed. **Patch: upgrade to vm2 3.11.4.** ([NVD CVE-2026-47131](https://nvd.nist.gov/vuln/detail/CVE-2026-47131); [NVD CVE-2026-47137](https://nvd.nist.gov/vuln/detail/CVE-2026-47137); [NVD CVE-2026-47208](https://nvd.nist.gov/vuln/detail/CVE-2026-47208); [NVD CVE-2026-47210](https://nvd.nist.gov/vuln/detail/CVE-2026-47210))

## 📄 ML research — second arXiv batch

Second-batch papers from the **2026-06-11 US-time submission window** (arXiv batches roll at 20:00 ET, so this is today's batch). Excludes the six IDs cited in this morning's Overview (`2606.13426`, `.13603`, `.13614`, `.13629`, `.13649`, `.13657`). All `[preprint]`, not peer-reviewed.

- **`2606.13610` (submitted 11 Jun 2026) — "One Polluted Page Is Enough."** A security/anomaly result for the LLM era: search-augmented LLMs (models that retrieve live web pages before answering, the engine behind AI "recommendations") can be turned into unwitting shills by *poisoning a single retrieved page*. The authors build **FORGE**, a benchmark that rewrites real products inside retrieved pages with fake reviews/promo copy, and measure how readily the model then recommends the fake. Why it matters: it reframes prompt-injection from "trick the chatbot" to "trick the *web the chatbot reads*," a supply-side attack on every retrieval-augmented assistant. [abstract](https://arxiv.org/abs/2606.13610) `[preprint]`

- **`2606.13680` (11 Jun 2026) — RA-RFT: learning to reason by analogy.** Standard RAG (retrieval-augmented generation — fetch relevant text, then answer) retrieves by surface similarity, which is wrong for reasoning: a problem that *looks* similar may need a totally different method, and vice-versa. **RA-RFT** post-trains the *retriever* with reinforcement fine-tuning and "gold-relevance distillation" to rank contexts by *expected reasoning benefit* rather than semantic closeness — i.e. retrieve the example that teaches the right *strategy*. A distinct paper from the June 7 RL-translation work on the same broad theme; treat as its own line. Why it matters: it attacks RAG's weakest link for math/logic tasks, where "similar wording" actively misleads. [abstract](https://arxiv.org/abs/2606.13680) `[preprint]`

- **`2606.13624` (11 Jun 2026) — adaptive token compression for time-series LLMs.** When you feed a language model both numbers (a sensor trace) and words (a prompt), it wastes compute treating every token alike. The authors show time-series tokens have "highly uneven spectral contributions" — many carry redundant frequency content while a few hold the critical temporal evidence — and that prompt-token influence fades with model depth. They compress asymmetrically: squeeze the redundant tokens, preserve the informative ones. Why it matters: efficient inference for the fast-growing "LLM as time-series forecaster" niche (energy, finance, ops monitoring), where context windows fill up with mostly-redundant numeric history. [abstract](https://arxiv.org/abs/2606.13624) `[preprint]`

- **`2606.13637` (11 Jun 2026) — The Stable Recovery Manifold.** An interpretability angle on catastrophic forgetting (when a network trained on task B loses task A). Training a ResNet-18 sequentially on Split CIFAR-100, the authors measure "Recovery Subspace Dimensionality" — the minimum number of singular directions you must restore to recover 90% of a forgotten task's probe accuracy — and find it stays *stable* across ten tasks, contradicting their own "recoverability diffuses" hypothesis. Why it matters: it suggests forgotten knowledge isn't smeared irretrievably across the weights but lives in a low-dimensional, recoverable subspace — which would make cheap "rehydration" of old skills plausible rather than requiring full retraining. [abstract](https://arxiv.org/abs/2606.13637) `[preprint]`

- **`2606.13634` (11 Jun 2026) — Operads for compositional reasoning in LLMs.** A pure-math framing of a practical trick. Question decomposition (break a hard query into sub-questions, answer each, recombine) works empirically but has no theory. The authors propose **operads** — algebraic structures from category theory that model "many-inputs, one-output" operations and how they compose — as the right scaffold: they define a "questions operad" where operations are question *templates* and composition is substituting sub-answers, and interpret a QA model as an *algebra* over it. The concrete anchor: think of an operad as a recipe grammar — each operation is a recipe step taking several ingredients to one dish, and composition is nesting recipes inside recipes; the claim is that LLM decomposition *is* such a grammar and can be reasoned about formally (when decomposition is valid, where it breaks). What's genuinely new is the formalism and the algebra-over-operad reading; whether it yields better decomposition in practice is left open. [abstract](https://arxiv.org/abs/2606.13634) `[preprint]`

- **`2606.13598` (11 Jun 2026) — OrchRM: reward modeling for multi-agent orchestration.** In multi-agent LLM systems an "orchestrator" routes work among specialist agents, but training it is expensive because you normally re-run the whole sub-agent pipeline to score a decision. **OrchRM** trains a reward model *at the orchestration level* using a Bradley-Terry win/lose formulation (the same pairwise-preference math behind RLHF reward models) over intermediate execution artifacts — no human labels, no costly sub-agent rollouts. Why it matters: cheaper, label-free orchestrator training is a practical lever for the agent-framework wave, where the routing decision often matters more than any single agent's output. [abstract](https://arxiv.org/abs/2606.13598) `[preprint]`
