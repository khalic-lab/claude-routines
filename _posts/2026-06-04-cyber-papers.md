## 🛡️ Cybersecurity

- **CVE-2026-41283 — OpenStack Mistral: unauthenticated remote code execution, CVSS 9.9 (critical).** OpenStack is the dominant open-source cloud-infrastructure platform; *Mistral* is its workflow-as-a-service component (it runs multi-step automation jobs). Versions through 22.0.0 expose API endpoints that let an attacker with *low* privileges run arbitrary code on the server — and because the CVSS vector is `AV:N/AC:L/PR:L/UI:N/S:C` (network-reachable, low complexity, no user interaction, and **scope-changed**, meaning the compromise spills beyond Mistral into the host), a single exposed Mistral API can become full control of the cloud control plane. The fix landed via the OpenStack security advisory **OSSA-2026-020**; operators should patch and ensure Mistral's API is never internet-exposed. Why it matters: this is the orchestration brain of a private cloud — RCE here (RCE = remote code execution, running attacker code on the target) is about as bad as cloud bugs get. ([NVD: CVE-2026-41283](https://nvd.nist.gov/vuln/detail/CVE-2026-41283); advisory [OSSA-2026-020](https://security.openstack.org/ossa/OSSA-2026-020.html))

- **CVE-2026-4104 — TeknoPass: authorization bypass via user-controlled SQL primary key, CVSS 9.8 (critical).** TeknoPass (a Turkish identity/access product) lets an *unauthenticated* attacker bypass authorization by tampering with a database primary key passed in a request — a *SQL injection* (SQLi: feeding crafted input into a database query so the query does something it shouldn't) that doubles as an auth-bypass. With `PR:N` (no privileges needed) and full confidentiality/integrity/availability impact, it is trivially weaponisable against any exposed instance. Disclosed through Turkey's national cyber authority USOM. Why it matters: unauthenticated SQLi in an *access-control* product is the worst place to have it — the thing guarding the door is the thing that breaks. ([NVD: CVE-2026-4104](https://nvd.nist.gov/vuln/detail/CVE-2026-4104); advisory [USOM TR-26-0309](https://siberguvenlik.gov.tr/guvenlik-bildirimleri/detay/tr-26-0309))

- **CVE-2026-10840 — Red Hat OpenShift Pipelines (Tekton): privilege escalation, CVSS 9.6 (critical).** OpenShift Pipelines is Red Hat's CI/CD system (continuous integration / continuous delivery — the automation that builds and ships software), built on the open-source *Tekton* project. A misconfigured `tekton-scheduler-rolebinding` ClusterRoleBinding grants the `system:authenticated` group — i.e. *any* authenticated user in the cluster — write access it should never have. The vector is `C:N/I:H/A:H` (no data-confidentiality loss, but high integrity and availability impact) with scope-change, so a low-privileged user can tamper with the build pipeline. Why it matters: write access to CI/CD is a textbook *software-supply-chain* foothold — poison the pipeline and you poison every artifact it ships. ([NVD: CVE-2026-10840](https://nvd.nist.gov/vuln/detail/CVE-2026-10840); [Red Hat advisory](https://access.redhat.com/security/cve/CVE-2026-10840))

- **CVE-2026-41010 / CVE-2026-41011 — Cloud Foundry / BOSH: command injection via package and job names, CVSS 8.7 (high).** Cloud Foundry is an open-source application-platform (PaaS) and *BOSH* is its deployment/lifecycle engine. ## 📄 ML research — second arXiv batch

_From today's arXiv announcement (papers submitted 4 Jun 2026, announced 2026-06-04), excluding the six IDs already cited in this morning's Overview. All `[preprint]`, not peer-reviewed._

- **`arXiv:2606.04284` (sub. 4 Jun 2026) — "Sparse Mixture-of-Experts Reward Models Learn Interpretable and Specialized Experts for Personalized Preference Modeling."** Most reward models in *RLHF* (reinforcement learning from human feedback — the step where a model is tuned to match human preference judgments) assume one universal notion of "good," ignoring that people disagree. This paper builds the reward model itself as a *sparse mixture-of-experts* (MoE: many small expert sub-networks, only a few of which fire per input), and finds the experts specialise into human-readable preference dimensions (e.g. helpfulness vs. safety vs. verbosity) you can then mix per user — without extra preference annotation. Why interesting: it makes the reward model both *personalisable* and *interpretable*, turning the usual opaque scalar reward into legible components. `[preprint]` ([abstract](https://arxiv.org/abs/2606.04284))

- **`arXiv:2606.04503` (sub. 4 Jun 2026) — "Smart Picks in the Dark: Towards Efficient RLVR for Reasoning via Tracing Metacognitive Pivots."** *RLVR* (reinforcement learning with verifiable rewards — training reasoning models on tasks where an answer can be automatically checked right/wrong, e.g. maths) is powerful but data-hungry. Existing fixes either need a pre-labelled "golden" subset or run fully unsupervised and underperform. This work selects which unlabelled problems to train on by tracing *metacognitive pivots* — the points in a chain of reasoning (chain-of-thought: the model's step-by-step working) where the model changes course — as a signal of which samples are most informative. Why interesting: a cheaper route to the reasoning gains of RLVR without a hand-labelled dataset, which is the main cost driver. `[preprint]` ([abstract](https://arxiv.org/abs/2606.04503))

- **`arXiv:2606.04050` (sub. 4 Jun 2026) — "LiftQuant: Continuous Bit-Width LLM via Dimensional Lifting and Projection."** *Quantization* shrinks a model by storing weights at low precision (e.g. 2- or 3-bit integers instead of 16-bit floats), cutting memory and speeding inference — but integer bit-widths are rigid, so a model rarely fits a given memory budget exactly. LiftQuant introduces *continuous* effective bit-widths: it projects a simple 1-bit lattice from a higher-dimensional "lifted" space, so you can dial precision to, say, 2.4 bits and land on the true Pareto-optimal point (the best accuracy achievable for that exact memory budget). Why interesting: it removes an artificial constraint, letting deployment hit a memory target precisely instead of rounding to the next whole bit. `[preprint]` ([abstract](https://arxiv.org/abs/2606.04050))

- **`arXiv:2606.04678` (sub. 4 Jun 2026) — "Test-Time Compute Scaling for ASR with Depth-Conditioned Looped Transformers."** In *ASR* (automatic speech recognition — turning audio into text), the acoustic encoder has a fixed depth, so the only way to spend more compute for better accuracy is usually to train a bigger model. This paper's **LARM** reuses one shared Transformer block recurrently — a *looped* architecture (same weights applied for several passes to add effective depth) — but, finding naive looping wastes the extra compute, conditions each pass on its depth (via FiLM-style modulation and sparse intermediate checkpoints) so recurrent depth becomes a clean, dial-at-inference compute knob. Why interesting: it ports the LLM idea of "think longer to do better" into speech recognition, trading test-time compute for accuracy *without* retraining a larger encoder. `[preprint]` ([abstract](https://arxiv.org/abs/2606.04678))

- **`arXiv:2606.04634` (sub. 4 Jun 2026) — "Explainably Safe Reinforcement Learning."** *Shielding* is a leading way to make RL agents safe: a formally-synthesised "shield" overrides any action that could violate a safety constraint. The catch is that shields, built by rigorous formal methods, are themselves opaque — you get the guarantee but not the *why*. This paper represents the shield (and policy) as *decision trees* (a transparent if-then structure a human can read) so the safety logic becomes inspectable while keeping the formal guarantee. Why interesting: it attacks the usual safety-vs-interpretability trade-off head-on — trustworthy autonomous systems arguably need both a proof *and* an explanation, and this aims to deliver them together. `[preprint]` ([abstract](https://arxiv.org/abs/2606.04634))
