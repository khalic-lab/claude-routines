## 🛡️ Cybersecurity

- **CVE-2026-12537 — Google Gemini CLI: CVSS 10.0, OS command injection to host RCE in CI.** A flaw in the container launcher of Google's Gemini CLI (versions before 0.39.1) and the companion `run-gemini-cli` GitHub Action (before 0.1.22) lets an unprivileged attacker achieve **pre-sandbox, host-level code execution** on headless CI runners via a maliciously crafted `.gemini/.env` file. "Pre-sandbox" is the dangerous part: the malicious command runs on the CI host *before* the tool's container isolation is established, so the sandbox never gets a chance to contain it. RCE = remote code execution (running attacker-chosen code on the target). CVSS 10.0 is the maximum (critical; trivially exploitable, full compromise). Patch: upgrade to Gemini CLI ≥ 0.39.1 / action ≥ 0.1.22. Concretely relevant for anyone running AI coding agents inside their build pipelines. ([NVD, 2026-06-24](https://nvd.nist.gov/vuln/detail/CVE-2026-12537); [GitHub Security Advisory GHSA-wpqr-6v78-jr5g](https://github.com/google-github-actions/run-gemini-cli/security/advisories/GHSA-wpqr-6v78-jr5g))

- **CVE-2026-56121 — Feast ML feature store: CVSS 9.3, unauthenticated RCE via `dill.loads()`.** Feast (a popular open-source feature store — the system that serves precomputed model inputs to ML training and inference) before 0.63.0 takes the `user_defined_function.body` field of an on-demand feature view, base64-decodes it, and passes it straight to `dill.loads()` — *before any authorization check*. `dill` is an extended Python pickle library, and deserialising untrusted pickle data is the classic Python RCE primitive: an attacker embeds an object with a malicious `__reduce__` method (the hook Python calls during unpickling) and ships it in a crafted gRPC request to the registry server, running arbitrary code with no credentials. This is a textbook insecure-deserialisation bug sitting on an unauthenticated network endpoint. Patch: upgrade to Feast ≥ 0.63.0. ([NVD, 2026-06-24](https://nvd.nist.gov/vuln/detail/CVE-2026-56121); [feast v0.63.0 release](https://github.com/feast-dev/feast/releases/tag/v0.63.0); [huntr report](https://huntr.com/bounties/d64b8111-180b-46ba-afa3-c877fda2ede6))

- **GeoVision GV-I/O Box 4E — a cluster of CVSS-10 command-injection and overflow flaws (Cisco Talos).** Cisco Talos disclosed a batch of critical vulnerabilities in the GeoVision GV-I/O Box 4E, a small networked industrial controller (4 inputs / 4 relay outputs over Ethernet and RS-485). The core problem sits in `DVRSearch`, a service that listens by default for UDP packets on port 10001: any host on the network can send it messages, and several lead to unauthenticated OS command injection (e.g. CVE-2026-12846, CVE-2026-12847, CVE-2026-12848 at CVSS 10.0) plus stack-buffer overflows from the way it copies up to 1460 bytes of attacker data. The pattern — an exposed UDP listener that hands network input to a shell — is exactly the IoT/OT (operational-technology) weakness that turns a single reachable device into a foothold on a control network. Verify patch availability with GeoVision before exposing these boxes. ([NVD CVE-2026-12846, 2026-06-24](https://nvd.nist.gov/vuln/detail/CVE-2026-12846); [Cisco Talos TALOS-2026-2377](https://talosintelligence.com/vulnerability_reports/TALOS-2026-2377))

- **CISA KEV additions: three Ubiquiti UniFi OS flaws now flagged as actively exploited.** CISA added three Ubiquiti UniFi OS vulnerabilities to its Known Exploited Vulnerabilities catalogue (KEV — the canonical list of bugs with confirmed in-the-wild exploitation, and a mandatory-patch list for US federal agencies) on 23 June: an improper-access-control flaw (CVE-2026-34908), a path-traversal flaw (CVE-2026-34909), and an improper-input-validation flaw (CVE-2026-34910). ([NVD CVE-2026-34908, 2026-06-23](https://nvd.nist.gov/vuln/detail/CVE-2026-34908); [CISA KEV catalog](https://www.cisa.gov/known-exploited-vulnerabilities-catalog))

- **CVE-2026-7574 — Anthropic Claude Desktop "Cowork": local persistence via unverified VM image.** A local-privilege issue (CVSS 8.7, local vector) in the Cowork virtual-machine feature of Anthropic's Claude Desktop app: across the confirmed range v1.1348.0 through v1.2278.0, the launcher checks only that the VM root-filesystem image (`rootfs.img`) is present and carries a version-marker string, but does **not** verify the image's content integrity at time-of-use. A local attacker already running unprivileged code as the victim macOS user can therefore modify the VM root filesystem and have the tampered image trusted on subsequent boots — turning a one-time foothold into persistent code execution inside the dev VM. It is not remotely exploitable (the attacker must already be on the machine), but it is a clean example of a time-of-check/time-of-use trust gap: validating that a file exists is not the same as validating that its contents are what you signed for. `[single-source]` ([NVD, 2026-06-24](https://nvd.nist.gov/vuln/detail/CVE-2026-7574))

## 📄 ML research — second arXiv batch

_All five papers below are from today's announcement batch (submitted 2026-06-23 US Eastern, announced 2026-06-24); none overlap the arXiv IDs cited in this morning's Overview brief._

- **arXiv:2606.24506 — CrossPool: stop reserving worst-case memory for "cold" MoE models.** _(submitted 2026-06-23)_ Why interesting: services that host many sparse Mixture-of-Experts models (MoE — models where each token activates only a few of many expert sub-networks) waste GPU memory because each model reserves capacity for its worst-case KV-cache (the cached attention keys/values that grow with conversation length), even though most "cold" models rarely hit peak demand together. CrossPool *disaggregates* the two kinds of memory — stable, model-determined weights versus transient, demand-determined KV-cache — into separate pools, so a shared KV pool can be provisioned for *aggregate* active demand rather than per-model worst case. The payoff is straightforward serving economics: fit more models on the same hardware. `[preprint]` ([abstract](https://arxiv.org/abs/2606.24506))

- **arXiv:2606.24143 — AsyncOPD: how stale can on-policy distillation be?** _(submitted 2026-06-23)_ Why interesting: on-policy distillation (OPD) trains a small "student" model on its *own* generated outputs, graded by a larger "teacher" — and, like reinforcement learning, it is bottlenecked by how long those rollouts take to generate. The standard fix is to run rollout-generation and weight-updates asynchronously, but that feeds the learner *stale* data (rollouts produced by a slightly older version of the student). This is the first systematic study of how much staleness OPD can tolerate before quality degrades — the practical knob that decides how aggressively you can parallelise post-training. `[preprint]` ([abstract](https://arxiv.org/abs/2606.24143))

- **arXiv:2606.24790 — Grad Detect: catching LLM hallucinations from the gradients, not the output.** _(submitted 2026-06-23)_ Why interesting: most hallucination detectors look only at what the model *said* (the output text or its probabilities). Grad Detect instead reads the model's internal **gradients** — the per-layer signals from a single forward-and-backward pass at inference time — and argues this internal structure carries information about whether an answer is correct that output-level signals miss. If it holds up, it is a cheap, white-box reliability check (one extra backward pass) for deploying LLMs in high-stakes settings. `[preprint]` ([abstract](https://arxiv.org/abs/2606.24790))

- **arXiv:2606.24543 — Reasoning as attractor dynamics: a physics-flavoured view of LLM reasoning.** _(submitted 2026-06-23)_ Why interesting: it reframes an LLM not as a left-to-right text generator but as a *dense associative memory* — a system that stores patterns as low-energy "attractor" states and settles into them, like a ball rolling into a valley. The conjecture: correct reasoning chains sit in deep, wide valleys ("flat minima" — stable, broad basins), while hallucinations are sharp, unstable dips. It then retrieves better answers by sampling several reasoning paths and weighting them by a Gibbs measure of their spectral entropy (borrowing the statistical-physics rule that lower-energy states are exponentially more likely). Honest scope: single-author preprint, conceptual framing on math reasoning rather than a large empirical win — interesting as a lens, not yet a method to rely on. `[preprint]` `[single-source]` ([abstract](https://arxiv.org/abs/2606.24543))

- **arXiv:2606.24133 — Holistic Data Scheduler: pick the pre-training data mix with multi-objective RL.** _(submitted 2026-06-23)_ Why interesting: the *mixture* of data sources (web, code, math, books…) is one of the biggest levers on what a model becomes, and "online data mixing" adjusts that blend adaptively during training. The authors argue existing methods optimise a single objective and so miss that a good mix must trade off several goals at once; their Holistic Data Scheduler casts the choice as multi-objective reinforcement learning (optimising a Pareto set of competing objectives rather than one number). The pitch is a more principled answer to a question that today is largely tuned by hand and intuition. `[preprint]` ([abstract](https://arxiv.org/abs/2606.24133))
