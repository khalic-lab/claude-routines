## 🛡️ Cybersecurity

A weekend NVD batch heavy on AI/ML developer tooling. Note the recurring theme: the same ecosystem that the reader works in — model-serving, agent frameworks, pickle scanning, web crawlers for LLM ingestion — keeps shipping security-relevant defaults (hardcoded keys, unvalidated inputs, scanners that miss). CVSS = Common Vulnerability Scoring System, 0–10; ≥9.0 is "critical." All CVE data below was read directly from the NVD JSON 2.0 API (a direct fetch, not a snippet).

- **picklescan misses a fresh set of malicious-pickle gadgets — RCE that evades the scanner (CVSS 7.6 HIGH).** Python's `pickle` format executes code on load, so the ML world relies on scanners like **picklescan** (used in Hugging Face's malware checks) to flag dangerous model files before they run. A new cluster of CVEs shows picklescan failing to detect several crafted "gadgets" — `torch.utils._config_module.load_config` ([CVE-2025-71348](https://nvd.nist.gov/vuln/detail/CVE-2025-71348)), `timeit.timeit()` ([CVE-2025-71351](https://nvd.nist.gov/vuln/detail/CVE-2025-71351)), `idlelib` runcommand ([CVE-2025-71357](https://nvd.nist.gov/vuln/detail/CVE-2025-71357)), and `cProfile.runctx` ([CVE-2025-71378](https://nvd.nist.gov/vuln/detail/CVE-2025-71378)) — each invoked inside a pickle's `__reduce__` method so the payload runs during `pickle.load` while the scan reports clean. Fixed across picklescan 0.0.25–0.0.30. This is a distinct batch from yesterday's picklescan issue (CVE-2026-56304) — different bypass functions, not the same flaw. Why it matters: a green checkmark from a pickle scanner is not proof of safety; treat untrusted `.pkl`/`.bin` model weights as code, prefer safetensors, and update picklescan. [NVD: CVE-2025-71348](https://nvd.nist.gov/vuln/detail/CVE-2025-71348)

- **Crawl4AI ships a hardcoded default JWT signing key — full authentication bypass (CVE-2026-56265, CVSS 9.3 CRITICAL).** Crawl4AI is a popular open-source web crawler used to feed pages into LLM pipelines (RAG — retrieval-augmented generation — and dataset building). Versions before 0.8.7 ship its Docker API server with a **hardcoded default JWT signing key** (a JSON Web Token is a signed credential; if everyone shares the secret used to sign it, anyone can mint a valid one). An attacker who knows the public default key can forge tokens for any user and gain full access to protected endpoints. Why it matters: deployments that exposed the Crawl4AI API without overriding the key are trivially authenticatable by anyone who read the source; rotate the key and upgrade to ≥0.8.7. [NVD: CVE-2026-56265](https://nvd.nist.gov/vuln/detail/CVE-2026-56265)

- **Prefect 3.6.23: git-flag injection in the GitRepository storage class yields RCE (CVE-2026-5366, CVSS 9.9 CRITICAL).** Prefect is a widely-used Python workflow-orchestration framework. Its `GitRepository` storage class passes the user-controlled `commit_sha` straight into git commands **without validation and without the `--` separator** that tells git "everything after this is data, not a flag." That lets an attacker smuggle in arbitrary git flags such as `--upload-pack`, which git will execute as an external program; the `directories` parameter is similarly injectable during sparse-checkout. Any user able to create a deployment can therefore run code on the Prefect server. Why it matters: argument-injection (the missing `--`) is a classic and easily-overlooked bug class — CVSS 9.9 reflects near-trivial exploitation with full server compromise. Upgrade past 3.6.23. [NVD: CVE-2026-5366](https://nvd.nist.gov/vuln/detail/CVE-2026-5366)

- **vLLM: unvalidated multimodal embeddings can crash the server or corrupt memory (CVE-2026-56340, CVSS 8.7 HIGH).** vLLM is the de-facto high-throughput inference engine for serving LLMs. In versions ≥0.10.2 and <0.13.0, when the `prompt-embeds` feature is enabled, the multimodal-embeddings path doesn't validate sparse tensors — and because PyTorch disables sparse-tensor invariant checks by default, a request carrying **malformed (negative or out-of-bounds) tensor indices** can trigger crashes, resource exhaustion (denial of service), and potentially out-of-bounds writes (memory corruption). Notably this **continues CVE-2025-62164**, whose earlier fix only turned the feature off by default rather than fixing the root cause. Why it matters: a single crafted inference request can take down a model-serving node — gate `prompt-embeds`, validate inputs, and upgrade to ≥0.13.0. [NVD: CVE-2026-56340](https://nvd.nist.gov/vuln/detail/CVE-2026-56340)

- **libexpat integer-overflow cluster fixed in 2.8.2 — a parser embedded almost everywhere (CVSS up to 6.9 MEDIUM).** **libexpat** is the small C library that parses XML inside a huge range of software (Python's standard library, browsers, language runtimes, countless apps), so a flaw there has unusually broad reach. A batch of eight integer-overflow bugs was disclosed and fixed in **2.8.2**, spanning core parsing routines — `storeAtts` ([CVE-2026-56403](https://nvd.nist.gov/vuln/detail/CVE-2026-56403)), `addBinding` ([56404](https://nvd.nist.gov/vuln/detail/CVE-2026-56404)), `getAttributeId` ([56405](https://nvd.nist.gov/vuln/detail/CVE-2026-56405)), `XML_ParseBuffer` missing a guard that `XML_Parse` had ([56406](https://nvd.nist.gov/vuln/detail/CVE-2026-56406)), `doProlog`/entity handling ([56407](https://nvd.nist.gov/vuln/detail/CVE-2026-56407)), `copyString` ([56408](https://nvd.nist.gov/vuln/detail/CVE-2026-56408)) — plus two in the `xmlwf` tool. Integer overflow here means a length calculation wraps around to a small value, leading to undersized allocations and potential out-of-bounds access on attacker-controlled XML. Why it matters: the bug is low-glamour but the blast radius is large — patch via your OS/runtime's expat package once 2.8.2 lands downstream. [NVD: CVE-2026-56403](https://nvd.nist.gov/vuln/detail/CVE-2026-56403)

- **A cluster of signed Windows kernel-driver flaws in consumer disk tools — local privilege escalation / BYOVD fuel (CVSS 7.1 HIGH each).** Several partition/backup/imaging utilities ship signed kernel drivers with the same class of weakness — an unprivileged process can reach driver functionality it shouldn't, escalating to kernel privileges: AOMEI Partition Assistant `ampa10.sys` ([CVE-2026-12778](https://nvd.nist.gov/vuln/detail/CVE-2026-12778)), AOMEI Dynamic Disk Manager `ddmdrv.sys` ([12779](https://nvd.nist.gov/vuln/detail/CVE-2026-12779)), AOMEI Backupper `amwrtdrv.sys` ([12780](https://nvd.nist.gov/vuln/detail/CVE-2026-12780)), EaseUS Partition Master `epmntdrv.sys`/`EUEDKEPM.sys` ([12781](https://nvd.nist.gov/vuln/detail/CVE-2026-12781)/[12782](https://nvd.nist.gov/vuln/detail/CVE-2026-12782)), IM-Magic Partition Resizer `MDA_NTDRV.sys` ([12784](https://nvd.nist.gov/vuln/detail/CVE-2026-12784)), and UltraISO `bootpt64.sys` ([12786](https://nvd.nist.gov/vuln/detail/CVE-2026-12786)). Why it matters: such *signed* but vulnerable drivers are exactly what "**BYOVD**" attacks abuse — Bring Your Own Vulnerable Driver, where an attacker drops a legitimately-signed but buggy driver to get kernel-level code execution and disable security tooling. Even if you don't run these utilities, defenders should consider blocking the specific driver hashes via Microsoft's vulnerable-driver blocklist. [NVD: CVE-2026-12781](https://nvd.nist.gov/vuln/detail/CVE-2026-12781)
