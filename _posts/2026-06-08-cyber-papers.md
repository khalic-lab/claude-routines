## 🛡️ Cybersecurity

No new CISA KEV (Known Exploited Vulnerabilities — CISA's authoritative list of flaws confirmed under active attack) additions today; the most recent entry remains **CVE-2026-28318** (SolarWinds Serv-U resource exhaustion, added 5 Jun), which this brief covered on 6 Jun. The day's signal is in newly published CVEs — notably a cluster in LLM-tooling and in RPKI routing-security software. CVSS scores below are the published NVD base scores; "CVSS 9.x (critical)" means trivially reachable and high-impact, "8.x (high)" means serious but with some precondition.

- **Flowise: a missing authorization check turns the LLM-flow builder into remote code execution — CVE-2026-46442 (CVSS 9.4, critical).** Flowise is a popular drag-and-drop UI for wiring up *LLM* (large language model) "flows" / agents. Before v3.1.2, `POST /api/v1/node-custom-function` had no route-level authorization, so any authenticated user or API key could submit arbitrary JavaScript to the Custom-JS-Function node — i.e. *RCE* (remote code execution: running attacker-chosen code on the server). Patched in 3.1.2. It ships alongside a cluster of related Flowise auth-bypass flaws — **CVE-2026-46444** (CVSS 8.7, unauthenticated CRUD on the OpenAI Assistants vector store) and **CVE-2026-46479/46480** (cross-workspace evaluation/evaluator takeover via mass-assignment). Why it matters: AI-orchestration platforms are becoming production infrastructure, and "missing middleware on one route" is exactly the boring bug that yields full server compromise; no known in-the-wild exploitation yet. ([NVD: CVE-2026-46442](https://nvd.nist.gov/vuln/detail/CVE-2026-46442); [Flowise GHSA-9rvc-vf7m-pgm2](https://github.com/FlowiseAI/Flowise/security/advisories/GHSA-9rvc-vf7m-pgm2); [NVD: CVE-2026-46444](https://nvd.nist.gov/vuln/detail/CVE-2026-46444))
- **Four CVEs in Routinator, the RPKI validator that underpins BGP route-origin security — CVE-2026-49232/49233/49234/49235 (CVSS 8.2–8.7, high).** Some context: the internet's routing protocol *BGP* (Border Gateway Protocol — how networks announce which IP-address blocks they carry) has no built-in proof that an announcement is legitimate, which enables route hijacks. *RPKI* (Resource Public Key Infrastructure) fixes this by cryptographically binding address blocks to the networks authorised to originate them; a *Relying Party* validator like NLnet Labs' **Routinator** fetches and checks those signed records so routers can reject bogus routes. The new flaws are mostly availability/integrity bugs in that validator: a malicious server can crash Routinator via a crafted RRDP XML *DTD* (Document Type Definition — an XML schema declaration) (CVE-2026-49235) or by exhausting connection handling (49232); a crafted rsync module name does *path traversal* (escaping the intended directory with `..`) into the cache (49233); and a non-UTF-8 API query parameter crashes it (49234). Why it matters: if attackers can reliably knock out RPKI validators, networks may fall back to accepting unvalidated routes — quietly re-opening the door to BGP hijacks. Fixes are in NLnet Labs' advisories. ([NVD: CVE-2026-49235](https://nvd.nist.gov/vuln/detail/CVE-2026-49235); [NLnet Labs advisory — 49233 path traversal](https://www.nlnetlabs.nl/downloads/routinator/CVE-2026-49233.txt); [NVD: CVE-2026-49232](https://nvd.nist.gov/vuln/detail/CVE-2026-49232))
- **VMware Cloud Foundation Operations: multiple stored XSS — CVE-2026-41722/41723/41724 (CVSS 8.0, high).** *Stored XSS* (cross-site scripting: attacker-supplied script saved by the app and later run in another user's browser session) in Cloud Foundation Operations lets a user who can create policies, views or text-widgets inject scripts that perform administrative actions when a higher-privileged user views the affected page. Broadcom has published an advisory. Why it matters: in a virtualization management console, "XSS that runs admin actions" is effectively privilege escalation across the tenant — and these consoles sit at the centre of enterprise infrastructure. ([NVD: CVE-2026-41722](https://nvd.nist.gov/vuln/detail/CVE-2026-41722); [Broadcom advisory 37513](https://support.broadcom.com/web/ecx/support-content-notification/-/external/content/SecurityAdvisories/0/37513))
- **Apache Cordova InAppBrowser (iOS): web content can fire arbitrary native callbacks — CVE-2026-47430 (CVSS 9.5, critical).** *Cordova* is a framework for building mobile apps with web technologies wrapped in a native shell; its **InAppBrowser** plugin opens web pages inside the app. On iOS the plugin passes the `id` field from a `WKScriptMessage` straight to the native bridge with no format validation, so any web content loaded in the InAppBrowser can invoke any pending native callback — a serious sandbox-escape-style bug for apps that load third-party or untrusted web content. Disclosed via the Apache security list and oss-security. Why it matters: many production apps embed Cordova WebViews; this lets a malicious page reach into the native layer. ([NVD: CVE-2026-47430](https://nvd.nist.gov/vuln/detail/CVE-2026-47430); [Apache security thread](https://lists.apache.org/thread/sb539nss3b0545wnyt1pbh7zgwjvz2qq))
- **Checkmk monitoring: stored XSS and a URL-validation bypass — CVE-2026-7186 and CVE-2026-8833 (CVSS 8.5, high).** Two issues in the Checkmk monitoring platform: a dashboard URL-widget that stores a dangerous `javascript:` URI and runs it for other viewers (7186), and a URL-validation function that can be bypassed using HTML-encoded characters to inject `javascript:` URIs (8833). Both affect Checkmk <2.5.0p5, <2.4.0p31, <2.3.0p48 and all 2.2.0; vendor "werk" patches are out. Why it matters: monitoring servers are high-trust, widely-viewed dashboards, so stored XSS there gives an attacker reach into many operators' sessions. ([NVD: CVE-2026-8833](https://nvd.nist.gov/vuln/detail/CVE-2026-8833); [Checkmk werk 20002](https://checkmk.com/werk/20002); [NVD: CVE-2026-7186](https://nvd.nist.gov/vuln/detail/CVE-2026-7186))

## 📄 ML research — second arXiv batch

All six are from the **8 Jun announcement batch** (arXiv announces weekend submissions on Monday evening ET; submission dates 4–5 Jun shown per paper). Excludes the six IDs already covered in this morning's Overview.

- **arXiv:2606.07404 (submitted 5 Jun) — "Reversible Foundations: Training a 120B Sparse MoE through State-Preserving Scaling" (LightningLM).** A systems/experience report on growing a *sparse MoE* (mixture-of-experts — a model with many specialist sub-networks where only a few "experts" fire per token, so total parameters can be huge while compute stays modest) from a small dense "seed" up to 120B parameters in four stages, each larger model inheriting the trained weights of the previous one, all on a single 8-GPU node. The 120B model has 460 experts under top-12 routing but only ~5.9B *active* parameters per token (~5% of the 118.7B stored). Why interesting: it's a recipe for reaching frontier-scale parameter counts on modest hardware by reusing weights across scales rather than training each size from scratch — relevant to anyone priced out of large clusters. `[preprint]` ([abstract](https://arxiv.org/abs/2606.07404))
- **arXiv:2606.07414 (submitted 5 Jun) — "Sparsely gated tiny linear experts" (sgatlin).** Pushes MoE sparsity to its limit: shrink each expert to a *single neuron* and select a tiny fraction of many available neurons. The counterintuitive trick is *removing* the nonlinearity usually applied to experts, leaving a network of sparsely-gated linear neurons; in an iso-FLOP comparison it replaces transformer feed-forward layers while improving both compute efficiency and interpretability. Why interesting: a clean rethink of what an "expert" needs to be — and the claim that going extreme on sparsity buys interpretability for free is a useful lead for mechanistic-interpretability work. `[preprint]` ([abstract](https://arxiv.org/abs/2606.07414))
- **arXiv:2606.07082 (submitted 5 Jun) — "On the Geometry of On-Policy Distillation."** *On-policy distillation* (OPD) trains a student model on its own generated outputs graded by a teacher, increasingly used to sharpen LLM reasoning. The paper maps OPD's training trajectory in weight space and contrasts it with *SFT* (supervised fine-tuning — imitating fixed example answers) and *RLVR* (reinforcement learning with verifiable rewards — RL where a checkable signal, e.g. "did the code pass tests," gives the reward). It finds OPD sits in a "relaxed off-principal" regime — touching fewer weights than SFT, less constrained than RLVR — and rapidly locks into a narrow low-dimensional subspace. Why interesting: post-training method choice is mostly folklore; concrete geometric diagnostics of *how* OPD moves a model help explain when it helps and when it collapses diversity. `[preprint]` ([abstract](https://arxiv.org/abs/2606.07082))
- **arXiv:2606.06664 (submitted 4 Jun) — "Inside the Visual Mind: Neuroscience-Motivated Concept Circuits for Interpreting and Steering Vision Transformers" (ViSAE).** A *mechanistic interpretability* toolbox (reverse-engineering the internal computations of a network, not just probing its outputs) for *ViTs* (Vision Transformers — image models built on the transformer architecture). It adapts *SAEs* (sparse autoencoders — a tool that decomposes a model's tangled internal activations into many human-nameable "concept" features) to vision and assembles them into "concept circuits," giving control over concept coverage and a path to *steer* a ViT away from spurious cues. Why interesting: ViTs often make right predictions for wrong reasons (latching onto background, not the object); a scalable way to find and intervene on the responsible features is a safety-before-deployment lever. `[preprint]` ([abstract](https://arxiv.org/abs/2606.06664))
- **arXiv:2606.06746 (submitted 4 Jun) — "Performance Variation in Deep Reinforcement Learning."** Tackles a notorious *deep RL* (reinforcement learning — agents learning by trial-and-error reward) reproducibility problem: identically-configured agents often give wildly different results across random seeds, yet papers typically report only uncertainty in the *mean*, which hides the spread. The authors argue that misleads, and propose percentile-based statistics (a "min–max IPR" and run-wise percentile visualisation) that surface run-to-run variation honestly. Why interesting: a measurement-hygiene fix for a field where "our method beats baseline" frequently fails to replicate — the kind of unglamorous tooling that improves whether published gains are real. `[preprint]` ([abstract](https://arxiv.org/abs/2606.06746))
- **arXiv:2606.06976 (submitted 5 Jun) — "Exploring Agentic Tool-Calling Decisions via Uncertainty-Aligned Reinforcement Learning" (TRUST).** LLM agents frequently make bad tool-use calls — invoking a tool they shouldn't, or *hallucinating* a confident direct answer instead of looking something up — and errors compound over multi-step tasks. The authors observe that ordinary outcome-based RL actually *weakens* the model's uncertainty separation between right and wrong actions (breeding overconfident mistakes), and propose TRUST to align uncertainty with correctness so the agent knows when to reach for a tool. Why interesting: it targets *calibration of the decision to act*, not just the action — a more principled handle on the overconfidence that makes autonomous agents brittle. `[preprint]` ([abstract](https://arxiv.org/abs/2606.06976))
