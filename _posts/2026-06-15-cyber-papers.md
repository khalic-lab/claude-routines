## 🛡️ Cybersecurity

NVD published 87 CVEs dated 2026-06-15 (25 at CVSS ≥7.5). No *new* CISA KEV (Known Exploited Vulnerabilities — CISA's catalogue of bugs confirmed under active attack) entries were added today; the most recent additions remain Oracle PeopleSoft PeopleTools (CVE-2026-35273, added 12 Jun, already covered) and Ivanti Sentry (CVE-2026-10520, 11 Jun). Today's high-severity set is dominated by vendor-disclosed and WordPress-plugin flaws rather than mass-exploited zero-days — patch-window stuff, not fire-drill stuff. Highlights:

- **CVE-2026-9862 — Fortra BoKS Privileged Access Manager, unauthenticated OS command injection, CVSS 9.8 (critical).** Fortra's Core Privileged Access Manager (BoKS) — an enterprise PAM product that brokers and controls privileged/root access across Unix/Linux fleets — contains an OS command injection (where attacker-supplied input is run as a shell command) in its `boks_autoregisterd` service. A remote attacker with network access to the service can run commands with the service's privileges *during the autoregistration process*, i.e. **without authenticating**. CVSS 9.8 means trivially exploitable over the network for full compromise. Why it matters: a PAM is the crown-jewels controller — the system that *gates* admin access — so an unauth RCE in it is close to a worst-case blast radius. A companion bug, **CVE-2026-9863** (CVSS 7.5, OS command injection in the legacy tar-based client upgrade/patch tooling), lets a malicious or compromised legacy client run commands on the BoKS master. Patch availability: Fortra has issued advisories FI-2026-007 / FI-2026-008. No public exploitation reported yet. [NVD CVE-2026-9862](https://nvd.nist.gov/vuln/detail/CVE-2026-9862); [Fortra FI-2026-007](https://www.fortra.com/security/advisories/product-security/fi-2026-007); [NVD CVE-2026-9863](https://nvd.nist.gov/vuln/detail/CVE-2026-9863).

- **CVE-2026-5482 — Responsive FileManager unauthenticated file-upload → RCE, CVSS 9.3 (critical).** Responsive FileManager (an open-source web file-manager widget) lets an unauthenticated attacker upload files of *any* type and extension via the `dialog.php` endpoint, with no restriction — the classic path from "arbitrary upload" to remote code execution (upload a web shell, then call it). The catch that makes this worse: the project is **unmaintained** at the time of CVE assignment, and the flaw is present in the latest release (9.14.0), so there is no vendor patch — defenders must remove or wall off the component. Reported by Poland's CERT.pl. No confirmed in-the-wild exploitation noted, but unauth-upload-to-RCE on an abandoned web component is exactly the kind of thing opportunistic scanners weaponise. [NVD CVE-2026-5482](https://nvd.nist.gov/vuln/detail/CVE-2026-5482); [CERT.pl advisory](https://cert.pl/en/posts/2026/06/CVE-2026-5482).

- **CVE-2026-49757 — AshAuthentication (Elixir/Ash framework) OAuth/OIDC account takeover, CVSS 9.2 (critical).** AshAuthentication is the authentication library for the Ash framework in the Elixir ecosystem. Its OAuth2 and OIDC (OpenID Connect — the standard "log in with Google/etc." identity layer on top of OAuth2) strategies matched users **by email address** (an upsert on the email field). The consequence: an attacker who can obtain an OAuth/OIDC sign-in carrying a victim's email — even from a different or attacker-controlled identity provider — can be matched onto the victim's *local* account, i.e. **authentication bypass by spoofing → account takeover**. Why it matters: this is a textbook "trust-the-email-claim" identity flaw; any app wiring social/SSO login into AshAuthentication should patch immediately. Fixes are in commits on the `team-alembic/ash_authentication` repo. [NVD CVE-2026-49757](https://nvd.nist.gov/vuln/detail/CVE-2026-49757); [ErlEF CNA advisory](https://cna.erlef.org/cves/CVE-2026-49757.html).

- **CVE-2026-5079 — `multer` (Express.js file-upload middleware) denial-of-service, CVSS 7.5 (high).** `multer` — one of the most widely used Node.js/Express middlewares for handling `multipart/form-data` file uploads — is vulnerable to a DoS via *deeply nested field names*. Its `append-field` dependency parses bracket-notation field names (`a[b][c][d]…`) with **no limit on nesting depth**, so an attacker can craft a request that forces allocation of a deeply nested object and exhausts memory/CPU — knocking the service over with a single malicious upload form. Affected: multer 1.0.0–2.1.1 and 3.0.0-alpha.1. Why it matters: multer sits in front of countless Node web apps; this is a cheap, unauthenticated way to take a service offline. Update to a patched release per the GitHub Security Advisory. [NVD CVE-2026-5079](https://nvd.nist.gov/vuln/detail/CVE-2026-5079); [GHSA-72gw-mp4g-v24j](https://github.com/expressjs/multer/security/advisories/GHSA-72gw-mp4g-v24j).

- **CVE-2026-11860 — Quick.CMS PHP-object deserialization over plaintext HTTP, CVSS 7.5 (high).** Quick.CMS deserializes user-controlled data received over **plaintext HTTP** without integrity or authenticity checks. "Deserialization" turns a byte stream back into live program objects; doing it on untrusted input with no class restrictions lets an attacker tamper with serialized payloads in transit (a man-in-the-middle on the unencrypted channel) and inject malicious objects — a classic deserialization-gadget path that can escalate toward code execution. Reported by CERT.pl. Why it matters: it's a double failure — no transport encryption *and* unsafe deserialization — so even a network-position attacker (not just the client) can exploit it. [NVD CVE-2026-11860](https://nvd.nist.gov/vuln/detail/CVE-2026-11860); [CERT.pl advisory](https://cert.pl/posts/2026/06/CVE-2026-11860/).

## 📄 ML research — second arXiv batch

Today's arXiv batch (announced for 2026-06-15; IDs `2606.13xxx–2606.14xxx`). All `[preprint]`. Papers already cited in this morning's Overview (2606.13705, .13753, .13740, .13767, .13754, .13685) are excluded. Selected from 249 new (announce-type "new") submissions across cs.LG / cs.AI / cs.CL / cs.CV / stat.ML.

- **arXiv:2606.14179 (15 Jun) — *CacheRL: Multi-Turn Tool-Calling Agents via Cached Rollouts and Hybrid Reward.*** Trains small "agent" models (LLMs that call external tools/APIs over multiple turns) to **92% process accuracy** on multi-step tool-calling — approaching GPT-5's reported 94% — while using **~100× less compute**. The trick: instead of running expensive *live* tool calls during reinforcement learning (RL — training by trial-and-error against a reward), it replays **cached rollouts** (recorded past tool interactions) and adds a hybrid reward to learn robustly from those noisy recordings. Why interesting: live tool execution is the dominant cost and flakiness source in agent RL; caching it could make capable tool-using agents trainable on modest budgets — a practical lever for anyone who can't afford GPT-5-scale runs. [abstract](https://arxiv.org/abs/2606.14179)

- **arXiv:2606.13862 (15 Jun) — *SuperThoughts: Reasoning Tokens in Superposition.*** Long chain-of-thought (CoT — the model "thinking out loud" step by step) is powerful but slow because tokens are generated one at a time. SuperThoughts compresses *pairs* of consecutive CoT tokens into a single latent (continuous, non-text) representation and decodes two tokens at once — roughly halving the sequential steps in the reasoning trace. Unlike prior "reason in latent space" attempts that were unstable and didn't scale, the authors claim a supervision signal that keeps training stable on long-horizon tasks. Why interesting: it attacks the core cost of reasoning models — the length of the thought chain — without abandoning the supervision that makes CoT trainable; if it holds up, it's a route to cheaper long reasoning. [abstract](https://arxiv.org/abs/2606.13862)

- **arXiv:2606.14150 (15 Jun) — *Small LLMs: Pruning vs. Training from Scratch.*** A controlled, token-matched study of whether *pruning* (cutting weights/layers out of a big trained model) beats just training a small model from scratch. Pruning Llama-3.1-8B at 50–80% removal with six methods (depth, width, and sparsity granularities), the finding: **under the same training-token budget, a pruned starting point consistently beats random initialization** — the parent model is a genuinely strong head start — but the advantage *narrows* as you pour in more training tokens. Why interesting: it puts numbers on a much-asserted folk belief ("just prune a big model") and clarifies *when* it pays off (token-constrained regimes) versus when training fresh catches up — useful guidance for anyone building small, cheap-to-run models. [abstract](https://arxiv.org/abs/2606.14150)

- **arXiv:2606.13982 (15 Jun) — *Adaptive Nucleus Truncation for Long-Form Reasoning.*** Over thousands of decoding steps, tiny changes in which candidate tokens are allowed compound into different reasoning paths and final answers. Existing samplers (top-_p_ / "nucleus", min-_p_, top-_nσ_ — all ways of pruning the model's next-token shortlist before sampling) use **fixed thresholds** that can't adapt to changing entropy (uncertainty) or task difficulty across a long generation. This paper makes the truncation **adaptive** — the cutoff responds to entropy, difficulty, and generation budget step by step. Why interesting: sampling settings are an underrated knob in reasoning models; an adaptive truncation that stabilises long chains could improve answer reliability without retraining anything — a free-ish win at inference time. [abstract](https://arxiv.org/abs/2606.13982)

- **arXiv:2606.14129 (15 Jun) — *BoRAD: Bootstrap your Own Representations for Multi-class Anomaly Detection.*** Anomaly detection (flagging the rare "this doesn't belong" — here, defects in industrial inspection) usually trains one model *per product category*. BoRAD targets the harder **one-for-all** setting: a single label-free model that reconstructs many normal appearances at once. That setting has two coupled failure modes the paper names — the *identical shortcut* (anomalies sneak through the reconstruction unchanged, so nothing looks wrong) and *mis-reconstruction* (normal categories get confused for each other). BoRAD's self-bootstrapped representations aim to break both. Why interesting: a unified, label-free detector that generalises across product types is far cheaper to deploy on a factory line than dozens of per-category models — the practical bottleneck in real inspection systems. [abstract](https://arxiv.org/abs/2606.14129)

- **arXiv:2606.13815 (15 Jun) — *Poker Arena: Multi-Axis Profiling of Strategic Reasoning and Memory in LLMs.*** Most game-playing LLM benchmarks collapse everything into one win/loss number, hiding *what kind* of reasoning a model is (or isn't) doing. Poker Arena runs a no-limit Texas Hold'em tournament with a three-layer memory (within-hand, within-session, across-session) and scores models on a **nine-axis cognitive profile** — decomposing strategic reasoning under uncertainty (bluffing, opponent modelling, bankroll/risk management) rather than reporting a single score. Why interesting: poker is a clean testbed for decision-making under hidden information and bluffing (negotiation, finance, policy all share that shape); a multi-axis profile shows *where* frontier models are strong or brittle, which a single leaderboard number can't. [abstract](https://arxiv.org/abs/2606.13815)
