## 🚀 New models, datasets, & open weights

- **DeepSeek ships "DSpark" — a faster way to *run* DeepSeek-V4 — and open-sources the
  DeepSpec codebase behind it (MIT).** Overnight (the new Hugging Face checkpoints are
  stamped 2026-06-27 03:02 UTC) DeepSeek published **DSpark-enhanced versions of both
  DeepSeek-V4-Pro and DeepSeek-V4-Flash**, plus **DeepSpec**, a full-stack open-source
  toolkit for training and evaluating the small "draft" models that power *speculative
  decoding*. Speculative decoding is a now-standard speed trick: a small, cheap **draft
  model** guesses the next several tokens, and the big target model checks them all in a
  single parallel pass — every guess it accepts is a token generated without paying for a
  full forward pass, so output gets faster with *no change to what the model says* (the
  big model still has final say on every token). DeepSeek's model card is explicit that
  this is an add-on, not a retrain: "DeepSeek-V4-Pro-DSpark is **not a new model. It is the
  same checkpoint with an additional speculative decoding module attached.**" The underlying
  DeepSeek-V4-Pro is a 1.6-trillion-parameter mixture-of-experts model (MoE — only ~49B
  parameters fire per token, so it runs far cheaper than its headline size) with a
  one-million-token context window; the public checkpoint ships in fp8 (8-bit) weights.
  DeepSpec is MIT-licensed and ships three interchangeable drafters — **DSpark, DFlash and
  Eagle3** — so the recipe is reusable on other models, not locked to DeepSeek's.
  Why it matters: the frontier open weights mostly come from Chinese labs now, and the
  bottleneck for actually using them is inference cost; releasing the *serving-efficiency*
  layer as open source — not just the weights — is how that ecosystem keeps compounding.
  ([DeepSpec, GitHub (DeepSeek)](https://github.com/deepseek-ai/DeepSpec); [model card, Hugging Face, 2026-06-27](https://huggingface.co/deepseek-ai/DeepSeek-V4-Pro-DSpark); V4 architecture per [DeepSeek-V4 tech report, arXiv:2606.19348](https://arxiv.org/abs/2606.19348) `[preprint]`; Flash variant: [DeepSeek-V4-Flash-DSpark](https://huggingface.co/deepseek-ai/DeepSeek-V4-Flash-DSpark))

- **What's actually new in DSpark: a "semi-autoregressive" drafter.** The interesting
  bit is the draft model's shape. Purely *parallel* drafters (which propose a whole block
  of tokens at once) are fast but get sloppy toward the end of the block — the later a
  guessed token sits, the less context it had, so the big model rejects more of them
  ("acceptance decay"). Fully *sequential* drafters are accurate but slow. DSpark splits
  the difference: a heavy parallel head proposes the block, then a small sequential module
  models the dependencies *within* the block to clean up those late-position guesses —
  hence "semi-autoregressive." More accepted guesses per pass = more speed-up. DeepSeek
  says DSpark has already been running in its production traffic for V4-Pro and V4-Flash.
  ([DeepSpec, GitHub (DeepSeek)](https://github.com/deepseek-ai/DeepSpec))

## 📊 Benchmarks & evaluations

- **DSpark's headline numbers: faster acceptance than Eagle3 and DFlash — but read them as
  vendor self-report.** Per DeepSeek's announcement (reached only through secondary coverage
  — the DSpark technical PDF in the repo didn't render through the proxy), DSpark lifts
  **"acceptance length" by roughly 16.3%–30.9%** over the two leading open drafters,
  **Eagle3** and **DFlash**. Acceptance length is the natural metric here: it's the average
  number of draft tokens the big model accepts per round — the higher it is, the fewer
  expensive full passes you pay for, so it maps almost directly to speed-up. DeepSeek's own
  evaluation harness in DeepSpec runs the standard spread — GSM8K and MATH-500 (grade-school
  and competition math), AIME'25, HumanEval and MBPP (code), LiveCodeBench, MT-Bench and
  Arena-Hard-v2 (chat) — which is a reasonable, public set. The caveat is the usual one for
  a method shipped by its own authors: the comparison was run by DeepSeek, on DeepSeek's
  models, and hasn't been independently reproduced. Treat the acceptance-length edge as a
  credible-but-unverified in-house result until a third party re-runs it.
  ([DeepSpec, GitHub (DeepSeek)](https://github.com/deepseek-ai/DeepSpec); numbers via [36Kr, 2026-06-27](https://eu.36kr.com/en/p/3871135542416645) `[via snippet]` `[single-source]`)

- **The end-to-end speed claim is real but reported inconsistently — so hold it loosely.**
  Secondary outlets put DSpark's throughput gain anywhere from **"up to 80%"** faster
  generation (the figure most coverage led with) to a much wider **"51% to 400%"** range
  cited from social posts. Those aren't contradictory so much as different conditions —
  speculative-decoding speed-up depends heavily on batch size, sequence length and how
  predictable the text is — but the spread is exactly why a single "N% faster" number is
  near-meaningless without the setup behind it. The honest read: DSpark meaningfully speeds
  up DeepSeek-V4 serving; the precise multiplier depends on your workload and isn't pinned
  down by a public, independent benchmark yet. ([DeepSeek V4 DSpark, KuCoin newsflash, 2026-06-27](https://www.kucoin.com/news/flash/deepseek-v4-launches-dspark-boosts-inference-speed-by-80) `[via snippet]` `[single-source]`)
