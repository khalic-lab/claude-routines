# fetch-proxy — Cloudflare Worker (page fetcher)

A small Worker that fetches a public URL **from Cloudflare's edge with a real browser
User-Agent** and streams the body back. The news-brief routines call it at compose time so
they can read lab blogs and news pages that otherwise return **HTTP 403** to the routine
sandbox.

Why it exists — the sandbox has two stacked problems:
1. **Allowlist egress only.** It can only reach the hosts in the `env_018` allowed-domains
   list. Enumerating every lab/news domain there is endless.
2. **Datacenter-IP + `curl` UA.** Even allowlisted, Cloudflare/Akamai-fronted sites (Anthropic,
   OpenAI, DeepMind, Meta, Mistral blogs; CNBC, TechCrunch, Bloomberg, Fortune, …) 403 the
   sandbox on sight as a bot.

This Worker collapses both: the sandbox only needs **one** allowlisted host
(`fetch-proxy.<account>.workers.dev`, same `*.workers.dev` pattern already allowed for
`embed-proxy`), and the actual fetch leaves from a Cloudflare edge IP with browser headers.

Twin of `tools/og-proxy` + `tools/embed-proxy`. Same account, same `wrangler` flow.

> **Scope / etiquette.** Low-volume personal reading of *public* pages. It does **not** defeat
> logins, paywalls, or JS/Turnstile challenges — sites behind hard Cloudflare Bot-Management
> may still 403 even through the Worker (it can't run their challenge JS). Keep arXiv on its
> direct allowlisted path (`export.arxiv.org`) — arXiv asks automated clients to use it and
> rate-limit, which we honor rather than route around.

## API

```
GET https://fetch-proxy.<account>.workers.dev/?url=<urlencoded target URL>
Authorization: Bearer <FETCH_TOKEN>
```

Response: the upstream body, with the upstream `Content-Type` and — importantly — the
**upstream HTTP status mirrored**. So a writer's `curl -fsSL` still *fails* (and falls back to a
search snippet) when the origin genuinely blocks us; the proxy never fakes a `200`. Extra
response headers: `X-Proxy-Upstream-Status`, `X-Proxy-Final-Url` (after redirects).

Errors generated by the proxy itself:
- `401` — missing/wrong bearer token (the host is public; the token gates abuse).
- `400` — missing/invalid `?url=`, or a non-http(s) target.
- `403` — target host is internal/private (SSRF guard: localhost, RFC1918, link-local/metadata).
- `405` — non-GET.
- `502` — upstream fetch failed, timed out (20 s), or body exceeded 5 MB.

## Deploy (one-time, ~3 minutes)

From this directory (`tools/fetch-proxy`):

1. `wrangler login` (skip if already logged in for og-proxy/embed-proxy — same account).
2. Set the shared secret (pick a long random string; reuse it in the writer prompts):
   ```
   wrangler secret put FETCH_TOKEN
   ```
3. Deploy:
   ```
   wrangler deploy
   ```
   Wrangler prints the live URL, e.g. `https://fetch-proxy.your-account.workers.dev`.
4. Send that URL back to the assistant. It then:
   - adds the **one** host to the `env_018` routine allowlist,
   - wires the URL + token into the writer prompts (curl-through-proxy before direct curl).

## Verify

```bash
WORKER=https://fetch-proxy.<account>.workers.dev
TOKEN=<the FETCH_TOKEN you set>

# Happy path — a page that 403s the sandbox directly should now come back 200 with HTML.
curl -s -o /dev/null -w "%{http_code}\n" -G "$WORKER/" \
  --data-urlencode "url=https://www.anthropic.com/news" \
  -H "Authorization: Bearer $TOKEN"
# expect: 200 (or the upstream's real status if it still hard-blocks)

# No / wrong token -> 401.
curl -s -o /dev/null -w "%{http_code}\n" -G "$WORKER/" --data-urlencode "url=https://example.com"

# SSRF guard -> 403.
curl -s -o /dev/null -w "%{http_code}\n" -G "$WORKER/" \
  --data-urlencode "url=http://169.254.169.254/latest/meta-data/" \
  -H "Authorization: Bearer $TOKEN"
```

Live logs while testing: `wrangler tail`.

## How the writers call it

The fetch step becomes "proxy first, direct curl as the fallback for already-allowlisted feeds":

```bash
# proxy fetch (works for any host, browser UA, CF edge IP):
curl -fsSL -G "$FETCH_PROXY/" --data-urlencode "url=$TARGET" -H "Authorization: Bearer $FETCH_TOKEN"
```

arXiv/NVD/CISA/Nature/Quanta/SRF/Le Temps/Al Jazeera/Semantic Scholar stay on **direct** curl
(they're allowlisted and feed-friendly); everything else (lab blogs, tech-news HTML) goes
through the proxy.
